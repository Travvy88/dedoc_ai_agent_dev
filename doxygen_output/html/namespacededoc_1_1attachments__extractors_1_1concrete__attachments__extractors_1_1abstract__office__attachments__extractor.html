\par \b Module Contract\/ \:\par \b Purpose\/ \: Provide the abstract foundation for Microsoft Office OpenXML format (docx, xlsx, pptx) attachment extractors: OLE stream parsing, zip-based media/embedding extraction.

\par \b Scope\/ \: Office-format attachment extraction: ZIP archive traversal, OLE binary stream parsing (Ole10Native, CONTENTS for PDF), media/embedding folder filtering. \par \b Input\/ \: File path, binary OLE stream content, zip archive entries. \par \b Output\/ \: List of AttachedFile objects (images, PDFs, OLE-native files) extracted from the Office archive. \par \b Links\/ \: [USES_API(9): olefile.OleFileIO, charset_normalizer.from_bytes] \par \b Links\/ \: [USES_API(8): AbstractAttachmentsExtractor._content2attach_file] \par \b Links\/ \: [USES_API(7): dedoc.utils.parameter_utils.get_param_need_content_analysis] \par \b Invariants\/ \:

  • _get_attachments ALWAYS returns List[AttachedFile] (may be empty).
  • .emf and .wmf files are always skipped (Windows metafile, not useful downstream).
  • .bin OLE files are parsed for embedded PDF or Ole10Native streams. \par \b Rationale\/ \:

Q: Why abstract instead of concrete? A: docx, xlsx, and pptx share identical ZIP+OLE attachment extraction logic; differences are only in the media folder path (word/, xl/, ppt/) passed by subclasses. Q: Why manual OLE binary parsing instead of a library? A: The Ole10Native binary format is simple (fixed-offset structure); a specialized library adds dependency weight without benefit. \par \b Changes\/ \:
