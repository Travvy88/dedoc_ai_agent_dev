protected

\par \b Module Contract\/ \:\par \b Purpose\/ \: To provide a flat, engine-agnostic canonical data model for OCR output — OCRWord, OCRLine, OCRResult — replacing the 5-level Tesseract-specific OcrPage hierarchy.

\par \b Scope\/ \: OCR result representation: word-level confidence+bbox, line-level grouping, annotation generation for dedoc pipeline. \par \b Input\/ \: Engine-specific dict/native output (converted by adapter). \par \b Output\/ \: List[Annotation] via get_annotations() for dedoc document pipeline. \par \b Links\/ \: [USES_API(8): dedoc.data_structures.annotation.*; USES_API(8): dedoc.data_structures.concrete_annotations.*; USES_API(9): dedocutils.data_structures.BBox] \par \b Invariants\/ \:

  • OCRWord ALWAYS has text (str, may be empty), bbox (BBox), and confidence (float 0-100).
  • OCRLine ALWAYS has non-None words list and bbox.
  • OCRResult ALWAYS has non-None lines list.
  • get_annotations() ALWAYS returns List[Annotation] matching old OcrLine.get_annotations() semantics. \par \b Rationale\/ \:

Q: Why flat dataclasses instead of the old 5-level OcrPage hierarchy? A: Any OCR engine can emit a list of words with bbox and confidence — no hierarchy needed. Flat dataclasses reduce coupling and simplify adapter writing for new engines (EasyOCR, PaddleOCR). \par \b Changes\/ \:

LAST_CHANGE: [v2.6.2 – Initial creation: OCRWord, OCRLine, OCRResult canonical dataclasses] \par \b Module Map\/ \:

CLASS 3[Single recognized word with bbox and confidence] => OCRWord CLASS 5[Group of words forming a text line with get_annotations()] => OCRLine CLASS 2[Container for OCRLines] => OCRResult \par \b Use Cases\/ \:

  • [OCRResult]: OCREngine → RecognizeImage → CanonicalDataModel → dedoc Pipeline
