\par \b Module Contract\/ \:\par \b Purpose\/ \: To define the abstract OCREngine contract and provide a factory function for creating OCR engine instances by name — enabling decoupled multi-engine OCR without hardcoding Tesseract.

\par \b Scope\/ \: OCR engine abstraction: interface definition (recognize_page, recognize_cells), factory for instantiation. \par \b Input\/ \: Engine name (str) and configuration dict. \par \b Output\/ \: OCREngine subclass instance. \par \b Links\/ \: [USES_TYPE(9): OCRResult, OCRLine, OCRWord; INSTANTIATES(8): TesseractEngine] \par \b Invariants\/ \:

  • get_ocr_engine("tesseract", config) ALWAYS returns a TesseractEngine instance.
  • get_ocr_engine(unknown_name, config) ALWAYS raises ValueError.
  • OCREngine subclasses MUST implement both recognize_page() and recognize_cells(). \par \b Rationale\/ \:

Q: Why abstract base class + factory instead of a simple callable? A: Strategy Pattern allows swapping OCR backends without touching orchestration code. Factory centralizes engine registration — new engines just add themselves to the registry. \par \b Changes\/ \:

LAST_CHANGE: [v2.6.2 – Initial creation: OCREngine ABC + get_ocr_engine() factory] \par \b Module Map\/ \:

CLASS 10[Abstract base class for OCR engines] => OCREngine FUNC 10[Factory returns OCREngine instance by name] => get_ocr_engine \par \b Use Cases\/ \:

  • [get_ocr_engine]: PdfImageReader (Init) → CreateEngine("tesseract") → TesseractEngine ready
  • [OCREngine]: Developer → ImplementNewEngine → RegisterInFactory → New engine available
